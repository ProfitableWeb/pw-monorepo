# ADR-003: Файловое хранилище

**Дата**: 2026-02-16

## Статус

Принято

## Контекст

Проекту нужно хранить пользовательские файлы: аватарки, изображения статей. Варианты: S3-совместимое облачное хранилище
(Cloud.ru Object Storage), локальное хранилище на VM.

Текущий масштаб: одна VM, один разработчик, десятки пользователей, ~40 статей.

## Решение

Локальное файловое хранилище на VM с минимальной абстракцией `StorageBackend`.

**Зачем абстракция**: весь код проекта работает с файлами через интерфейс `StorageBackend` (save, delete, url). Сейчас
за ним стоит `LocalStorage` — запись на диск VM. Когда появится prod-сервер на отдельной VM, файлы нужно будет хранить в
общем месте (S3). Абстракция позволит заменить `LocalStorage` на `S3Storage` в одну строку конфига, не трогая остальной
код.

**Что это НЕ означает**: S3 не нужен сейчас. Абстракция — это 3 метода в одном файле, а не фреймворк.

### Структура на диске

```
~/profitableweb/uploads/
├── avatars/{user_id}.webp          # 1:1 с пользователями
└── articles/{article_id}/          # группировка по статье
    ├── cover.webp
    └── img-001.webp
```

### Раздача файлов

Nginx отдаёт `/uploads/` как статику — без нагрузки на Python:

```nginx
location /uploads/ {
    alias /home/webresearcher/profitableweb/uploads/;
    expires 30d;
    add_header Cache-Control "public, immutable";
}
```

### Обработка изображений

```
upload → /tmp/pw-{uuid}.jpg     # принял файл
       → Pillow resize + WebP   # обработал (256x256 для аватарок)
       → uploads/avatars/...    # сохранил на диск
       → удалил /tmp файл
```

### Абстракция StorageBackend

```python
# apps/api/src/services/storage.py
# ADR-003: абстракция для замены LocalStorage → S3Storage при появлении prod VM

class StorageBackend(ABC):
    """3 метода — весь интерфейс работы с файлами в проекте."""
    async def save(self, path: str, data: bytes) -> str: ...   # → URL файла
    async def delete(self, path: str) -> None: ...
    def url(self, path: str) -> str: ...                       # path → публичный URL

class LocalStorage(StorageBackend):
    """Сейчас: сохраняет в uploads/ на диске, nginx отдаёт как статику."""

# class S3Storage(StorageBackend):
#     """Будущее: когда появится prod VM и нужно общее хранилище.
#     Триггеры для перехода — см. ADR-003, секция «Миграция на S3»."""
```

Весь остальной код (эндпоинты, сервисы) вызывает `storage.save()` / `storage.url()` — ему безразлично, куда сохраняется
файл.

## Почему не иерархия по датам

- Аватарки: ключ = `user_id`, один файл на пользователя, не растёт бесконтрольно
- Статьи: группировка по `article_id` — при удалении статьи удаляется вся папка
- Хронология (`/YYYY/MM/`) затрудняет очистку: непонятно какие файлы связаны с каким контентом

## Почему не шардинг по хешу

- ext4 с dir_index: O(1) lookup по имени файла через HTree
- Деградация листинга (`ls`) начинается при 100K+ файлов — недостижимо для этого проекта
- Шардинг (`ab/cd/abcd1234.webp`) усложняет структуру без пользы

## Альтернативы

| Альтернатива                 | Плюсы                 | Минусы                                     | Почему отклонена          |
| ---------------------------- | --------------------- | ------------------------------------------ | ------------------------- |
| Cloud.ru Object Storage (S3) | Бэкапы, CDN, безлимит | Доп. стоимость, сетевые задержки, overhead | Overkill для одной VM     |
| БД (BLOB)                    | Всё в одном месте     | Раздувает БД, медленная отдача             | Антипаттерн для файлов    |
| Иерархия /YYYY/MM/           | Привычная структура   | Сложно чистить при удалении контента       | Нет связи файл → сущность |

## Последствия

### Положительные

- Нулевая стоимость и латентность
- Простота разработки и отладки
- Nginx отдаёт статику максимально быстро

### Отрицательные

- Бэкапы файлов нужно настраивать отдельно (rsync/tar)
- При двух VM (test + prod) файлы не шарятся

### Миграция на S3

Триггеры для перехода:

- Появление prod VM (нужно общее хранилище)
- Объём файлов > 5 GB
- Потребность в CDN
