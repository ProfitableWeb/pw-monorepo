# ADR-001: База данных

**Дата**: 2026-02-10

## Статус

Принято

## Контекст

Проекту нужна реляционная БД для хранения пользователей, статей, категорий и комментариев. Варианты: управляемая БД в
облаке, PostgreSQL на VM, SQLite для простоты.

Бюджет ограничен — одна VM на Cloud.ru (213.171.25.187) для всех сервисов.

## Решение

PostgreSQL 14, установленный напрямую на VM (не managed/cloud-сервис).

### Детали реализации

- **ORM**: SQLAlchemy 2.0 с async-поддержкой (asyncpg)
- **Миграции**: Alembic с autogenerate
- **Подключение**: `DATABASE_URL` передаётся через CI/CD секреты в `.env`
- **Пользователь БД**: `profitableweb`, база: `profitableweb`
- **Seed-данные**: скрипт `uv run python -m src.seed` для начального наполнения

### Конфигурация

```bash
# На VM
sudo -u postgres createuser profitableweb
sudo -u postgres createdb profitableweb -O profitableweb
```

```
# .env
DATABASE_URL=postgresql+asyncpg://profitableweb:password@localhost:5432/profitableweb
```

## Альтернативы

| Альтернатива                | Плюсы                   | Минусы                                | Почему отклонена           |
| --------------------------- | ----------------------- | ------------------------------------- | -------------------------- |
| Cloud.ru Managed PostgreSQL | Бэкапы, масштабирование | Дополнительная стоимость ~2000₽/мес   | Бюджет ограничен           |
| SQLite                      | Простота, нет сервера   | Нет concurrent writes, нет async      | Не подходит для prod       |
| MongoDB                     | Гибкая схема            | Overkill для блога, нет строгих типов | Реляционная модель удобнее |

## Последствия

### Положительные

- Нулевая дополнительная стоимость (всё на одной VM)
- Полный контроль над конфигурацией
- Знакомый стек для разработки

### Отрицательные

- Бэкапы нужно настраивать вручную (cron + pg_dump)
- При росте нагрузки — миграция на managed-сервис

### Риски

- Потеря данных при сбое VM без бэкапов
