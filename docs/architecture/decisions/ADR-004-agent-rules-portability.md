# ADR-004: Переносимость агентных правил между AI-инструментами

**Дата**: 2026-05-03

## Статус

Принято

## Контекст

ProfitableWeb разрабатывается как агентно-ориентированный проект (см. `docs/agentic-development.md`). Агентный контекст
распределён по нескольким файлам и форматам:

- `AGENTS.md` (root) — короткий портативный слой
- `.claude/CLAUDE.md` — entrypoint для Claude Code
- `apps/admin/CLAUDE.md` — nested-контекст admin-приложения
- `.cursor/rules/*.mdc` — 17 файлов granular-правил Cursor с `globs` и `alwaysApply`
- `.claude/rules/{git-commit-format,task-creation}.md` — **дубликаты** содержимого `.cursor/rules/`
- `docs/agentic-development.md`, `docs/templates/`, `docs/architecture/`

Задача PW-069 — спроектировать переносимую схему до того, как в проекте появятся новые инструменты (Codex, Copilot,
OpenCode, Gemini, Windsurf), и устранить риск рассинхронизации.

### Какие инструменты целевые на первом этапе

| Инструмент                   | Что читает                                                                | Поддержка nested               | Filtering по globs     |
| ---------------------------- | ------------------------------------------------------------------------- | ------------------------------ | ---------------------- |
| **Cursor**                   | `.cursor/rules/*.mdc` (frontmatter `description`, `globs`, `alwaysApply`) | да (поддиректории)             | да                     |
| **Claude Code**              | `CLAUDE.md` по иерархии cwd→root + `@`-импорты                            | да                             | нет (всегда вся ветка) |
| **GitHub Copilot**           | `.github/copilot-instructions.md` (один файл)                             | нет                            | нет                    |
| **Codex (OpenAI)**           | `AGENTS.md` (root)                                                        | nested-AGENTS.md в подпроектах | нет                    |
| **OpenCode**                 | `AGENTS.md` + `.opencode/` плагины                                        | да (через AGENTS.md)           | нет                    |
| **Aider / Cline / Continue** | `AGENTS.md` или конвенциональный fallback                                 | частично                       | нет                    |
| **Windsurf**                 | `.windsurfrules` или `.windsurf/rules/*.md`                               | да                             | нет                    |
| **Gemini-CLI**               | `gemini.md` или `AGENTS.md` (зависит от версии)                           | частично                       | нет                    |

Ограничения проекта:

- Контент проекта — **на русском** (включая правила и комментарии).
- Гранулярные правила Cursor с `globs` экономят контекст и должны сохраниться.
- Sync-скрипт оправдан только при 3+ инструментах с реальной потребностью генерации — сейчас активны только Cursor и
  Claude Code.

## Решение

**Tiered shared + tool-specific без canonical `.agents/` и без sync-скрипта.**

`AGENTS.md` (root) — единственный портативный слой. Tool-specific форматы (`.cursor/rules/*.mdc`, `.claude/CLAUDE.md`,
`.github/copilot-instructions.md`) живут параллельно и **ссылаются** на `AGENTS.md` или на канонические Cursor-правила
вместо дублирования содержимого. `.claude/rules/*.md` — устранены, потому что повторяли `.cursor/rules/*.mdc`.

### Архитектура слоёв

```
Tier 0  (portable)
  AGENTS.md ← entrypoint для любого инструмента, читающего AGENTS.md
              (стек, команды, переносимые правила, карта контекста, workspace facts)

Tier 1  (tool-specific entrypoints)
  .claude/CLAUDE.md             ← Claude Code: обзор + ссылки на AGENTS.md и .cursor/rules/
  .cursor/rules/*.mdc           ← Cursor: granular правила с globs/alwaysApply (канонический источник деталей)
  .github/copilot-instructions.md ← Copilot: короткий pointer на AGENTS.md

Tier 2  (nested per-app)
  apps/admin/CLAUDE.md          ← Claude Code, специфика admin
  apps/web/CLAUDE.md            ← (добавлять по мере необходимости)
  apps/api/CLAUDE.md            ← (добавлять по мере необходимости)

Reference (общие источники)
  docs/agentic-development.md   ← принципы агентной работы, context headers
  docs/architecture/decisions/  ← ADR
  docs/architecture/runbooks/   ← операционные гайды
  docs/templates/               ← шаблоны README и context headers
```

### Что генерируется автоматически, что — вручную

| Элемент                         | Источник правды                   | Производные                                              |
| ------------------------------- | --------------------------------- | -------------------------------------------------------- |
| Granular tech-rules с `globs`   | `.cursor/rules/*.mdc`             | Не генерируются — Cursor-only формат                     |
| Портативные high-signal правила | `AGENTS.md`                       | Не генерируются — каждый инструмент читает напрямую      |
| Claude Code обзор               | `.claude/CLAUDE.md`               | Ссылается на `.cursor/rules/` через текстовые упоминания |
| Copilot инструкции              | `.github/copilot-instructions.md` | Pointer на AGENTS.md                                     |
| App-specific контекст           | `apps/*/CLAUDE.md`                | По необходимости — отдельный nested AGENTS.md            |

Sync-скрипт `scripts/sync-agent-rules` **не вводится** на этом этапе. Гипотетический триггер для его появления:

- 3+ инструмента, требующих идентичных правил в разных форматах
- регулярные расхождения между копиями (≥ 1 раза в квартал)
- автоматизация добавляет ценность сверх ручного редактирования двух-трёх файлов

### MCP memory / export / import

MCP-инструменты (`.mcp.json`, FastMCP, ресурсы `blog://*`) — **отдельный слой памяти и контекста**, не замена агентных
правил. MCP даёт runtime-инструменты для работы с контентом блога (CRUD статей, медиа, SEO), а не
архитектурные/процессные правила. Эти слои не пересекаются и не вытесняют друг друга.

## Альтернативы

| Альтернатива                                     | Плюсы                                         | Минусы                                                                                                   | Почему отклонена                                                             |
| ------------------------------------------------ | --------------------------------------------- | -------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| **Canonical `.agents/` + sync-скрипт**           | Один источник правды; защищает от расхождений | Накладные расходы на скрипт, генерация теряет filtering Cursor по `globs`, ломает сильную сторону `.mdc` | YAGNI: 2 активных инструмента, расхождений мало; теряем гранулярность Cursor |
| **Только `AGENTS.md`, удалить `.cursor/rules/`** | Максимальная простота                         | Cursor теряет filtering по `globs` — лишний контекст в каждом запросе; правила раздуваются               | Потеря инструмента-специфичной мощности                                      |
| **Только `.cursor/rules/`, без AGENTS.md**       | Гранулярность сохраняется                     | Codex/OpenCode/Aider не читают `.mdc`, теряют контекст полностью                                         | Не работает для не-Cursor инструментов                                       |
| **Дублировать всё в каждом формате вручную**     | Каждый инструмент получает родной формат      | Высокий риск рассинхронизации (уже наблюдалось с `.claude/rules/`)                                       | Не масштабируется                                                            |

## Последствия

### Положительные

- **Один источник правды** для каждого правила: переносимое — в `AGENTS.md`, гранулярное — в `.cursor/rules/`.
- **Cursor сохраняет** granular filtering по `globs`, экономит контекст.
- **Codex / OpenCode / Aider / Cline** получают рабочий entrypoint без правок.
- **Claude Code** работает по nested-CLAUDE.md без дублирования содержимого правил.
- **Copilot** покрыт минимальным pointer-файлом.
- Нет лишней автоматизации (sync-скрипта) — порог изменений = редактирование 1–2 файлов.

### Отрицательные

- При добавлении переносимого правила нужно решить: попадает ли оно в `AGENTS.md` (Tier 0) или остаётся
  Cursor-специфичным с `globs`. Это требует осознанности, но не автоматизации.
- Claude Code не загружает `.cursor/rules/*.mdc` автоматически — если требуется детальное правило, ссылку даёт
  `.claude/CLAUDE.md` (текстом).

### Риски

- При появлении 3-го инструмента, требующего собственного формата, придётся пересмотреть решение и, возможно, ввести
  canonical-схему. Граница пересмотра зафиксирована выше.
- Если `AGENTS.md` начнёт расти и дублировать `.claude/CLAUDE.md` — сигнал к разделению ответственности (`AGENTS.md`
  остаётся portable high-signal, `.claude/CLAUDE.md` — обзор и архитектура).
