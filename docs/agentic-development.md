# Агентно-ориентированная разработка

ProfitableWeb ведётся как проект для совместной работы человека и AI-кодинг-агентов. Это значит, что репозиторий должен
содержать достаточно контекста, чтобы агент понимал архитектуру, ограничения и намерения без длинного устного брифинга.

## Карта агентного контекста

ProfitableWeb использует **tiered shared + tool-specific** модель: один портативный слой + native-форматы для конкретных
инструментов. Архитектурное обоснование — `docs/architecture/decisions/ADR-004-agent-rules-portability.md`.

| Слой      | Файл / каталог                    | Кому                                         | Назначение                                                                  |
| --------- | --------------------------------- | -------------------------------------------- | --------------------------------------------------------------------------- |
| Tier 0    | `AGENTS.md` (root)                | любой инструмент, поддерживающий `AGENTS.md` | портативный entrypoint: стек, команды, переносимые правила, карта контекста |
| Tier 1    | `.claude/CLAUDE.md`               | Claude Code                                  | обзор проекта, архитектура, ключевые файлы                                  |
| Tier 1    | `.cursor/rules/*.mdc`             | Cursor                                       | granular-правила с `globs` и `alwaysApply` (канонический источник деталей)  |
| Tier 1    | `.github/copilot-instructions.md` | GitHub Copilot                               | короткий pointer на `AGENTS.md`                                             |
| Tier 2    | `apps/*/CLAUDE.md`                | Claude Code                                  | nested-контекст конкретного app                                             |
| Reference | `docs/architecture/`              | все                                          | ADR, runbooks, системная документация                                       |
| Reference | `docs/tasks/`                     | все                                          | постановки задач, KANBAN, история                                           |
| Reference | `docs/templates/`                 | все                                          | шаблоны README и context headers (frontend, backend, shared)                |

### Принципы переносимости

- **Один источник правды на правило**. Переносимое — в `AGENTS.md`, гранулярное технологическое — в
  `.cursor/rules/*.mdc`, app-specific — в `apps/*/CLAUDE.md`. Не дублировать то же содержимое в нескольких файлах.
- **Tool-specific форматы — это не дубликат, а адаптер**. `.cursor/rules/*.mdc` и `.github/copilot-instructions.md`
  существуют, потому что инструменты не читают друг друга. Дублируется только короткая ссылка, не содержание.
- **Sync-скрипт не вводится**, пока активны 1–2 инструмента. Порог пересмотра — 3+ инструмента и регулярные расхождения.
  Подробности — в ADR-004.
- **MCP — отдельный слой памяти и runtime-инструментов**, а не замена rules. Не переносить агентные правила в
  MCP-ресурсы.

### Поддерживаемые инструменты на первом этапе

| Инструмент                                  | Что читает                                        | Filtering по globs |
| ------------------------------------------- | ------------------------------------------------- | ------------------ |
| Cursor                                      | `.cursor/rules/*.mdc`                             | да                 |
| Claude Code                                 | `.claude/CLAUDE.md` + `apps/*/CLAUDE.md` (nested) | нет                |
| GitHub Copilot                              | `.github/copilot-instructions.md`                 | нет                |
| Codex / OpenCode / Aider / Cline / Continue | `AGENTS.md` (root)                                | нет                |

## Принципы

1. **Контекст рядом с кодом**. Важные решения должны быть в docs, правилах или коротких комментариях рядом с модулем.
2. **Короткие правила лучше больших**. Один `.mdc` — одна тема. Если правило растёт, его нужно дробить.
3. **Код важнее устаревшей инструкции**. Если правило или документ расходятся с реальным кодом, сначала проверить код,
   затем обновить инструкцию.
4. **Не документировать очевидное**. Комментарий должен добавлять информацию, которой нет в имени файла, функции или
   типе.
5. **Синхронность обязательна**. Комментарии и context headers обновляются в том же изменении, что и поведение.

## Context headers в коде

Для важных модулей добавляется короткая шапка-комментарий: справка для человека и агента о роли файла в системе.

На frontend это JSDoc:

```ts
/**
 * Контекст: зачем существует модуль и какую роль играет в фиче.
 * Границы: что модуль делает и чего принципиально не делает.
 * Связи: ключевые store/api/hooks/protocols, если без них файл трудно понять.
 */
```

Шаблон: `docs/templates/frontend/context-header.ts.md`.

На Python — module docstring:

```python
"""Контекст: роль модуля в backend.

Границы: за что модуль отвечает и какие слои не должен трогать.
Связи: сервисы, модели, внешние API или auth-flow, важные для понимания.
"""
```

Шаблон: `docs/templates/backend/context-header.py.md`.

### Где добавлять

- Orchestrator-компоненты страниц, секций и сложных виджетов.
- Zustand stores, React Context providers, API clients, query-key factories.
- Сложные hooks, протоколы `postMessage`, auth/permissions, MCP tools.
- Backend-сервисы с бизнес-правилами, интеграциями или нетривиальными транзакциями.
- Модули, где ошибка понимания приведёт к security, data loss или regressions.

### Где не добавлять

- Маленькие UI-примитивы, очевидные constants/types/utils.
- Файлы, где шапка повторяет имя компонента или сигнатуру функции.
- Автогенерируемый код и миграции, если комментарий не объясняет ручное решение.
- Комментарии вида “компонент отображает кнопку” — это шум, не контекст.

## Что писать в context header

- **Назначение**: зачем модуль существует в продукте.
- **Границы ответственности**: что здесь не должно появляться.
- **Связи**: какие store, API, события, cookies, query keys или внешние протоколы важны.
- **Инварианты**: правила, нарушение которых ломает поведение.
- **Причина нетривиального решения**: только если без неё будущий читатель “исправит” код неправильно.

## Почему аккуратно, а не везде

Исследования по комментариям сходятся в одном: качественные комментарии помогают пониманию, но плохие и устаревшие
вредят сильнее, чем отсутствие комментария.

- Eye-tracking study 2025 показывает, что комментарии направляют внимание, но эффект на performance зависит от сниппета:
  от ухудшения до улучшения. Вывод: важна полезность комментария, не сам факт наличия.
- Исследование function header documentation (`arXiv:2401.07704`) показывает, что header-docs ценны, когда дают
  информацию сверх сигнатуры: usage, edge cases, examples.
- Исследование LLM comprehension legacy code (`arXiv:2506.11007`) показывает, что точные комментарии помогают LLM
  понимать код, а неточные заметно ухудшают результат.
- Работы по comment smells и code-comment inconsistency показывают, что misleading/outdated comments ухудшают
  maintainability и повышают риск bug-introducing changes.

Практический вывод для ProfitableWeb: context header — это не ритуал, а маленький контракт понимания. Если он не
добавляет новой информации или его трудно поддерживать актуальным, его не нужно писать.

## Чеклист для агента

- Перед созданием нового context header проверь, не объясняет ли это уже имя файла, README секции или соседний документ.
- Если меняешь поведение модуля с context header, обнови header в том же diff.
- Если файл стал orchestrator или начал связывать несколько подсистем, добавь header.
- Если header разросся больше 8-10 строк, перенеси подробности в README/ADR и оставь ссылку.
- Не используй header для списка всех импортов или пересказа JSX/SQL/CRUD.
- Для README frontend/admin секций используй `docs/templates/frontend/section-readme.md`.
