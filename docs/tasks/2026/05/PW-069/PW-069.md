## Постановка задачи

### Описание

Сейчас ProfitableWeb использует несколько слоёв агентного контекста: `AGENTS.md`, `.claude/CLAUDE.md`,
`apps/*/CLAUDE.md`, `.cursor/rules/*.mdc`, `docs/agentic-development.md` и `docs/templates/`. Это работает, но при
подключении новых инструментов (Codex, Copilot, OpenCode, Gemini, Windsurf) может появиться рассинхронизация правил.

Нужно отдельно исследовать и спроектировать переносимую схему правил: оставить текущую модель или ввести canonical
source (`.agents/` или `docs/ai-rules-base.md`) и генерацию tool-specific adapters.

### Цель

Сформировать архитектуру multi-agent portability для ProfitableWeb без дублирования правил и без потери преимуществ
tool-specific форматов вроде `.cursor/rules/*.mdc`.

### Критерии приемки

- [ ] Зафиксировано, какие инструменты должны поддерживаться на первом этапе (Cursor, Claude Code, Codex, Copilot и др.)
- [ ] Описана роль `AGENTS.md`: общий portable слой, nested files или только root
- [ ] Принято решение: оставить shared + specific или вводить canonical `.agents/`
- [ ] Если выбран canonical source, спроектирован sync-скрипт для генерации `.cursor/rules`, Claude/Copilot/Codex
      adapters
- [ ] Описано, какие правила можно генерировать автоматически, а какие должны оставаться tool-specific
- [ ] Учтены MCP memory/export/import подходы только как отдельный слой памяти, не как замена rules
- [ ] Обновлён `docs/agentic-development.md` по итогам решения

### Зависимости

- PW-068 — агентно-ориентированная разработка и правила контекста.

### Техническое решение

Предварительная гипотеза:

- `AGENTS.md` — portable high-signal слой.
- `.cursor/rules/*.mdc` — granular Cursor layer с `globs`.
- `.claude/CLAUDE.md` — Claude Code entrypoint со ссылками на `AGENTS.md` и `.cursor/rules`.
- `.agents/` вводить только если появится sync-скрипт и реальная потребность поддерживать 3+ agent tools.
