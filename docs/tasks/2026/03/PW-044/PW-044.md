# PW-044: URL-роутинг для разделов админ-панели

**Статус**: TODO **Создано**: 2026-03-15 **Приоритет**: Normal **Компонент**: Admin

## Проблема

Сейчас навигация админки работает через Zustand-стор (`navigation-store.ts`) без изменения URL. При любом переходе между
разделами URL остаётся `/admin/`. Это значит:

- Нельзя скопировать ссылку на конкретный раздел
- Нельзя перезагрузить страницу без потери контекста (всегда возвращает на дашборд)
- Кнопки браузера "назад/вперёд" не работают
- Нельзя открыть конкретный раздел из закладок или по прямой ссылке

## Цель

Каждый раздел админки имеет отражение в URL. Навигация работает и через sidebar, и через адресную строку. Перезагрузка
страницы сохраняет текущий раздел. Кнопки browser history работают.

### Хабы (content-hub, editorial-hub)

`content-hub` и `editorial-hub` — обзорные страницы группы. Получают URL `/admin/content` и `/admin/editorial`.

Хабы доступны через клик по заголовку группы в sidebar (не отдельным пунктом). Реализовано: `hubId` в `NavSection`,
заголовок становится `<button>` при наличии hubId.

### Подразделы внутри секций

Некоторые секции имеют внутренние вкладки (users → список, команда, приглашения...; seo → основное, база знаний). На
первом этапе **подразделы НЕ отражаются в URL**. Это можно добавить позже как отдельную задачу.

Исключения — параметрические маршруты:

- `/admin/content/articles/:id` — редактор конкретной статьи
- `/admin/content/research/:id` — рабочее пространство исследования

## Техническое решение

### Подход: History API + Zustand sync (без новых зависимостей)

Не добавлять react-router-dom. Вместо этого — тонкий слой синхронизации URL ↔ Zustand:

```
User clicks sidebar → navigateTo(pageId) → Zustand update + history.pushState(url)
Browser back/forward → popstate event → parse URL → navigateTo(pageId, { skipPush: true })
Page load / F5 → parse URL → set initial currentPage
```

### Почему не react-router-dom

- Вся маршрутизация — плоский switch по `currentPage` в `App.tsx`
- Нет вложенных layouts, нет lazy loading по маршрутам
- 22 страницы — manageable без фреймворка
- Zustand-стор уже хранит всё нужное состояние (editArticleId, seoKbArticleId)
- Добавить router = переписать App.tsx, все `navigateTo` вызовы, интеграция с Zustand
- History API + popstate = ~50 строк кода

**Если в будущем** появятся вложенные layouts, code splitting по маршрутам или guard-middleware — тогда миграция на
react-router-dom будет оправдана. Но не сейчас.

записать полные пути с группой (`'content/articles'`). Одна точка правды, без дублирования в отдельном файле.

**Файл**: `apps/admin/src/app/lib/routes.ts` (новый) — только утилиты парсинга.

```typescript
/** Построить URL из PageId + опциональных параметров */
function pageIdToUrl(pageId: PageId, params?: Record<string, string>): string;

/** Распарсить pathname → PageId + параметры. null если маршрут не найден */
```

### PW-044-B: Синхронизация navigateTo → pushState

**Файл**: `apps/admin/src/app/store/navigation-store.ts`

Модифицировать `navigateTo`:

```typescript
navigateTo: (pageId: PageId, options?: { skipPush?: boolean }) => {
  set(state => {
    /* ... existing logic ... */
  });
  if (!options?.skipPush) {
    const url = pageIdToUrl(pageId);
    window.history.pushState({ pageId }, '', `/admin/${url}`);
  }
};
```

Аналогично для `navigateToArticleEditor` и `navigateToSeoKb` — они должны включать параметры в URL.

### PW-044-C: Восстановление при загрузке

В `App.tsx` или в инициализации стора — при первом рендере:

```typescript
// Парсим текущий URL и переходим на нужную страницу
const route = urlToPageId(window.location.pathname);
if (route) {
  navigateTo(route.pageId, { skipPush: true });
  if (route.params?.id) {
    // Установить editArticleId или researchId
  }
}
```

### PW-044-D: Обработка popstate

```typescript
useEffect(() => {
  const handler = (event: PopStateEvent) => {
    const pageId = event.state?.pageId || urlToPageId(location.pathname)?.pageId || 'dashboard';
    navigateTo(pageId, { skipPush: true });
  };
  window.addEventListener('popstate', handler);
  return () => window.removeEventListener('popstate', handler);
}, []);
```

### PW-044-E: Параметрические маршруты

Для `article-editor` и `research-workspace`:

```typescript
navigateToArticleEditor: (articleId?: string) => {
  // ... existing logic ...
  const url = articleId ? `content/articles/${articleId}` : 'content/articles/new';
  window.history.pushState({ pageId: 'article-editor', articleId }, '', `/admin/${url}`);
};
```

При парсинге URL `/admin/content/articles/uuid-xxx` → `{ pageId: 'article-editor', params: { id: 'uuid-xxx' } }`.

### PW-044-G: Кликабельные заголовки групп → хабы (DONE)

Заголовки групп "Контент" и "Редакция" в sidebar стали кликабельными — переход на content-hub / editorial-hub.
Реализовано через `hubId` в `NavSection` + замена `<h3>` на `<button>` для секций с hubId.

### PW-044-H: Редирект на intended URL после логина

Сценарий: неавторизованный пользователь открывает `/admin/system/users` → видит LoginPage → логинится → сейчас попадает
на дашборд, а должен на `/admin/system/users`.

Реализация: перед показом LoginPage сохранить `window.location.pathname` в переменную (ref или стор). После успешного
логина — `navigateTo` по сохранённому пути.

### PW-044-F: Проверка SPA fallback в nginx/Docker

SPA fallback (`try_files → index.html`) и host-level проксирование `/admin/*` **уже настроены** (PW-043). Подзадача
сводится к проверке, что всё работает с новыми вложенными путями:

- Открыть `/admin/system/users` напрямую → должен загрузиться SPA и показать раздел
- Открыть `/admin/content/articles/uuid` → SPA + редактор статьи
- Проверить, что ассеты (`/admin/assets/*`) не попадают под fallback
