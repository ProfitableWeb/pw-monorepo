# PW-040: Интеграция админ-панели с Article CRUD API

**Статус**: Выполнено **ID задачи**: PW-040 **Контекст**: ProfitableWeb Admin (`apps/admin`) **Зависит от**: PW-038
(бэкенд Article CRUD API), PW-037 (фронтенд Article Workbench)

## 1. Цель

Заменить моковые данные в Article Workbench реальными API-вызовами. Подключить все admin-эндпоинты (CRUD статей,
публикация, ревизии, теги, категории, системные настройки) к фронтенду через api-client и React Query.

## 2. Scope

### Включено

- Admin API-клиент: функции для всех 16 admin-эндпоинтов
- React Query: mutation-хуки (create, update, delete, publish, schedule, unpublish)
- React Query: query-хуки для admin-списка, admin-деталей, ревизий
- Замена mock-данных в ArticleWorkbench на реальные API-вызовы
- Маппинг полей `ArticleFormData` (camelCase) ↔ API (snake_case)
- Резолвинг category slug → UUID при сохранении
- Получение системного часового пояса из `/api/admin/settings`
- Интеграция истории ревизий (вкладка History)
- Автосохранение (debounced PATCH) с обновлением статуса в Zustand-сторе
- Query key invalidation после мутаций

### Исключено

- Создание новых UI-компонентов (интерфейс уже реализован в PW-037)
- Загрузка изображений / медиатека (image_url остаётся текстовым полем)
- AI-генерация контента / артефактов
- Управление пользователями / ролями
- Настройки (UI секции Settings) — только получение timezone для datepicker

## 6. Маппинг полей

### 6.1. API response → ArticleFormData (загрузка в редактор)

При получении `AdminArticleResponse` трансформация в `ArticleFormData` для react-hook-form:

```typescript
function apiToFormData(article: AdminArticleResponse, timezone: string): ArticleFormData {
  return {
    h1: article.title, // API title → форма h1
    subtitle: article.subtitle ?? '',
    title: article.metaTitle ?? '', // API meta_title → форма title (SEO)
    slug: article.slug,
    status: article.status as ArticleStatus,
    publishedAt: article.publishedAt ?? '',
    publishTimezone: timezone, // Из системных настроек, не из статьи
    category: article.category.slug, // Объект → slug для select
    tags: article.tags.map(t => t.name), // Объекты → имена
    excerpt: article.excerpt,
    imageUrl: article.imageUrl ?? undefined,
    content: article.content,
    // SEO
    metaDescription: article.metaDescription ?? '',
    canonicalUrl: article.canonicalUrl ?? '',
    ogTitle: article.ogTitle ?? '',
    ogDescription: article.ogDescription ?? '',
    ogImage: article.ogImage ?? '',
    focusKeyword: article.focusKeyword ?? '',
    seoKeywords: article.seoKeywords ?? [],
    robotsNoIndex: article.robotsNoIndex,
    robotsNoFollow: article.robotsNoFollow,
    schemaType: article.schemaType ?? 'BlogPosting',
    author: article.author?.name ?? '',
    artifacts: article.artifacts ?? defaultArtifacts,
```

### 6.2. ArticleFormData → API payload (сохранение)

При отправке PATCH/POST трансформация обратно. **Важно**: payload отправляется в snake_case.

```typescript
function formDataToPayload(form: ArticleFormData, categories: AdminCategory[]): ArticleUpdatePayload {
  // Резолвинг category slug → UUID
  const cat = categories.find(c => c.slug === form.category);

  return {
    title: form.h1, // Форма h1 → API title
    subtitle: form.subtitle || null,
    slug: form.slug,
    content: form.content,
    excerpt: form.excerpt,
    category_id: cat?.id, // slug → UUID
    tags: form.tags,
    image_url: form.imageUrl || null,
    // SEO
    meta_title: form.title || null, // Форма title (SEO) → API meta_title
    meta_description: form.metaDescription || null,
    canonical_url: form.canonicalUrl || null,
    og_title: form.ogTitle || null,
    og_description: form.ogDescription || null,
    og_image: form.ogImage || null,
    focus_keyword: form.focusKeyword || null,
    seo_keywords: form.seoKeywords,
    schema_type: form.schemaType,
    robots_no_index: form.robotsNoIndex,
    robots_no_follow: form.robotsNoFollow,
    artifacts: form.artifacts,
  };
}
```

## 7. Сценарии интеграции

### 7.1. Открытие существующей статьи

```
1. Пользователь кликает статью в Content Hub → навигация с article.id
2. useAdminArticle(articleId) — загрузка полных данных
3. useSystemSettings() — получение timezone
4. apiToFormData(article, settings.timezone) → defaultValues для react-hook-form
5. openArticle(slug, content) в Zustand store
6. Рендер ArticleWorkbench с реальными данными
```

### 7.2. Создание новой статьи

```
1. Кнопка «Новая статья» в Content Hub
2. Навигация в ArticleWorkbench без articleId
3. Пустая форма (defaultValues с дефолтами)
4. handleSave → createArticle mutation
5. После успеха: redirect на /admin/articles/{newId} (редактирование)
6. Invalidation adminArticleKeys.lists()
```

### 7.3. Сохранение (handleSave)

```
1. form.getValues() → ArticleFormData
2. formDataToPayload(formData, categories) → snake_case payload
3. Если новая статья → createArticle(payload)
   Если существующая → updateArticle(articleId, payload)
4. onSuccess:
   - toast «Сохранено»
   - markSaved() в Zustand store
   - invalidateQueries(adminArticleKeys.detail(id))
5. onError:
   - toast с ошибкой из API (detail)
   - setAutosaveStatus('offline') в Zustand
```

### 7.4. Автосохранение (debounced)

```
1. form.watch() подписка на изменения
2. Debounce 3-5 секунд
3. Если isDirty && articleId существует:
   - setAutosaveStatus('syncing')
   - updateArticle(articleId, changedFields)
   - onSuccess: markSaved(), setAutosaveStatus('saved')
   - onError: setAutosaveStatus('offline')
4. Отправляются только изменившиеся поля (partial update)
```

### 7.5. Публикация

```
1. Кнопка «Опубликовать» → publishArticle(articleId)
2. API валидирует: title, content, excerpt, category_id
3. Если 422 → показать ошибку «Для публикации необходимы: ...»
4. Если OK → обновить форму (status='published', publishedAt)
5. Invalidation: detail + lists
```

### 7.6. Планирование публикации

```
1. Выбор даты в datepicker
2. scheduleArticle(articleId, publishedAt: ISO string с timezone)
3. publishedAt должен быть AwareDatetime (с timezone offset)
4. API валидирует: дата в будущем + publishable fields
5. Если OK → status='scheduled', publishedAt обновлён
```

### 7.7. История ревизий (вкладка History)

```
1. useRevisions(articleId) — список ревизий
2. Отображение: дата, автор, summary
3. Клик → useRevision(articleId, revisionId) → просмотр content
4. Кнопка «Восстановить» → restoreRevision(articleId, revisionId)
5. После восстановления: обновить форму (content, contentFormat)
```

### 7.8. Удаление статьи

```
1. Кнопка «Удалить» → confirm dialog
2. deleteArticle(articleId, permanent=false) — мягкое удаление (→ archived)
3. Redirect в Content Hub
4. Invalidation: lists
```

## 11. Технические решения

### Payload в snake_case, response в camelCase

API-клиент уже имеет авто-маппинг snake→camel для ответов. Для запросов (POST/PATCH) payload формируется вручную в
snake_case — это проще, чем автоматическая конвертация, и позволяет контролировать маппинг `h1 → title`.

### Category slug → UUID

Фронтенд хранит `category` как slug (для удобства select-компонента). При сохранении нужен UUID. Решение: загрузить
список категорий через `useAdminCategories()`, найти UUID по slug при формировании payload. Список категорий кешируется
в React Query (staleTime: 5 min).

### Автосохранение — partial update

PATCH принимает только изменённые поля. Для автосохранения: отслеживать через `form.formState.dirtyFields`, отправлять
только dirty-поля. Это минимизирует трафик и конфликты при параллельном редактировании.

### Timezone — системная, не per-article

`publishTimezone` в `ArticleFormData` не сохраняется в API. Значение берётся из `GET /api/admin/settings` и
подставляется в форму при загрузке. При изменении системного timezone — все открытые редакторы обновятся через React
Query refetch.

### Артефакты — camelCase ↔ snake_case в JSON

Поле `artifacts` хранится как JSONB. Фронтенд использует camelCase (`selfCheck`, `workspaceId`), бэкенд принимает as-is.
Маппинг ключей внутри JSON делается в `lib/mappers.ts` при отправке и получении.

### articleId vs slug

Текущий Zustand store хранит `articleSlug`. Admin API работает по UUID (`articleId`). Нужно расширить store:
`articleId: string | null` (UUID для API-вызовов), `articleSlug` оставить для URL/навигации.
