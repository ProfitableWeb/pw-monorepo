## Архитектура OAuth-потока (решение из PW-046)

```
Админ → кнопка «Подключить» → popup (Яндекс OAuth)
  → Яндекс: авторизация приложения
  → redirect на callback-страницу с ?code=xxx
  → callback-страница: postMessage(code) → родительское окно
  → админка: POST /api/admin/seo/yandex/connect { code }
  → бэкенд: exchange code → token → encrypt → save to DB
  → ответ: { connected: true, account: "ivan@yandex.ru", permissions: [...] }
```

**Ключевые отличия от пользовательского OAuth (auth.py):**

- Это **не** вход пользователя — это подключение **сервисного токена** для всего сайта
- Один токен на систему, хранится в SystemSettings (не привязан к конкретному пользователю)
- Scope: `metrika:read webmaster:hostinfo` (не `login:info login:email`)
- Требуется отдельный Yandex OAuth App ID (или расширение текущего с дополнительными scopes)

### PW-048-B: OAuth-эндпоинты

**Файл**: `api/admin/seo.py` (расширить из PW-047)

```
GET  /api/admin/seo/yandex/auth-url → { url: "https://oauth.yandex.ru/authorize?..." }
POST /api/admin/seo/yandex/connect  { code: "xxx" } → { connected, account, permissions }
POST /api/admin/seo/yandex/disconnect → { connected: false }
```

**auth-url**: формирует URL с параметрами:

- `client_id` из Settings
- `response_type=code`
- `scope=metrika:read webmaster:hostinfo`
- `redirect_uri` = callback-страница в админке

**connect**: обменивает code на token (POST `https://oauth.yandex.ru/token`), получает info о пользователе, шифрует
токен, сохраняет в SystemSettings.

**disconnect**: удаляет токен и metadata из SystemSettings.

### PW-048-G: Фронтенд — OAuth-попап и реальные данные

**OAuth-попап:**

1. Callback-страница в админке: `apps/admin/src/app/pages/yandex-callback.html` (или компонент) — получает `code` из
   URL, отправляет `postMessage` родителю
