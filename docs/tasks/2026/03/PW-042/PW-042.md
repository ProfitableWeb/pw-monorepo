# PW-042: Логирование, мониторинг и аудит

**Статус**: DONE **Создано**: 2026-03-09 **Приоритет**: Normal **Компонент**: Backend + Admin + Infra

## Цель

Внедрить систему логирования, мониторинга и аудита — полностью свою, без внешних SaaS-зависимостей. Логи должны быть
структурированными и machine-readable — для ИИ-агентов, которые помогают в разработке и отладке.

## Контекст

### Текущее состояние — почти ноль

| Компонент              | Состояние                                              |
| ---------------------- | ------------------------------------------------------ |
| Python logging         | Один `logger.exception()` в OAuth — и всё              |
| Structured logging     | Нет (ни structlog, ни loguru)                          |
| Request logging        | Нет middleware, только uvicorn default access log      |
| Error tracking         | Нет — ошибки уходят в stdout PM2 и теряются            |
| Audit trail            | Нет — кто что делал в админке не фиксируется           |
| System monitoring      | Нет — ни `/health` эндпоинта, ни проверки диска/памяти |
| PM2 log rotation       | Не настроено — логи будут расти бесконечно             |
| nginx log rotation     | Системная (logrotate), не в репозитории                |
| Cloud.ru S3 мониторинг | Есть в панели Cloud.ru (API-запросы, объём, квоты)     |

### Принцип: всё своё, ничего лишнего

Мы живём в эпоху ИИ-автоматизации — целая система файлового хранилища создаётся за несколько часов. Задача по логам и
мониторингу примерно того же масштаба. Никаких внешних SaaS:

- **Нет Sentry** — своя таблица `error_log` в PostgreSQL + UI в админке
- **Нет UptimeRobot** — свой `/health` endpoint, при желании cron + Telegram бот
- **Нет ELK/Grafana** — structlog → stdout → PM2 → ротация; ошибки в БД → админка
- **Данные остаются у нас** — никаких внешних DSN, токенов, зависимостей от чужих серверов

## PW-042-A: Structured Logging

### Выбор: structlog

| Критерий                | stdlib logging  | loguru         | structlog            |
| ----------------------- | --------------- | -------------- | -------------------- |
| Структурированный JSON  | Через formatter | serialize=True | Нативный             |
| AI-agent readability    | Низкая          | Средняя        | **Высокая**          |
| Context binding         | LoggerAdapter   | contextualize  | bind() — first-class |
| Совместимость с uvicorn | Да              | Intercept      | Wraps stdlib         |
| Dev console             | Текст           | Цветной        | ConsoleRenderer      |

**structlog** — данные первичны, форматирование вторично. JSON в проде, цветной текст в dev.

### Что реализовать

1. **Конфигурация structlog** в `core/logging.py`:
   - Processor pipeline: timestamp, log level, request_id, user_id, JSON renderer (prod) / console (dev)
   - Перехват uvicorn и SQLAlchemy логов через stdlib bridge

2. **Request middleware** (`middleware/logging.py`):
   - `asgi-correlation-id` для X-Request-ID
   - Bind: request_id, method, path, client_ip, user_id
   - Log: `event="request_completed"`, status_code, duration_ms
   - **Не логировать**: тела запросов/ответов, cookies, авторизацию

3. **Event naming convention**:
   - Формат: `domain.action` — `article.published`, `media.uploaded`, `auth.login_failed`
   - Уровни: DEBUG (dev only), INFO (нормальные операции), WARNING (деградация), ERROR (сбой), CRITICAL (запуск
     невозможен)

### Зависимости

```
structlog
asgi-correlation-id
```

## PW-042-B: Error Tracking — своё

### Почему не Sentry

- Внешняя зависимость (DSN, токены, чужие сервера)
- Free tier имеет лимиты и может измениться
- У нас уже есть PostgreSQL и паттерн для админки — нужно только таблица + middleware + UI
- Для блога с одним разработчиком простая таблица ошибок даёт 90% ценности Sentry

### Модель `error_log`

```python
class ErrorLog(Base):
    __tablename__ = "error_log"

    id: Mapped[UUID]
    timestamp: Mapped[datetime]         # default=utcnow
    level: Mapped[str]                  # 'error', 'critical', 'warning'
    event: Mapped[str]                  # 'unhandled_exception', 's3_upload_failed'
    message: Mapped[str]                # Краткое описание
    traceback: Mapped[str | None]       # Полный traceback
    request_method: Mapped[str | None]  # GET, POST...
    request_path: Mapped[str | None]    # /api/admin/articles
    request_id: Mapped[str | None]      # Correlation ID
    user_id: Mapped[UUID | None]        # Кто столкнулся с ошибкой
    status_code: Mapped[int | None]     # 500, 422, etc.
    context: Mapped[dict | None]        # JSONB — дополнительные данные
    resolved: Mapped[bool]              # default=False, для «прочитано/исправлено»
```

### Перехват ошибок

1. **Exception middleware** — перехватывает все необработанные исключения, пишет в `error_log`
2. **Явный вызов** — `error_service.log(...)` в критических местах (S3, OAuth, миграции)
3. **Группировка** — по event + request_path (аналог Sentry issue grouping, но простой)

### API

```
GET  /admin/errors           → пагинированный список ошибок
GET  /admin/errors/{id}      → детали с traceback
POST /admin/errors/{id}/resolve → пометить как resolved
GET  /admin/errors/stats     → количество ошибок за 24ч / 7д / 30д
```

## PW-042-D: Audit Log

### Модель

```python
class AuditLog(Base):
    __tablename__ = "audit_log"

    id: Mapped[UUID]
    timestamp: Mapped[datetime]
    user_id: Mapped[UUID | None]     # FK -> users, nullable (системные действия)
    action: Mapped[str]              # 'article.create', 'media.upload', 'auth.login'
    resource_type: Mapped[str]       # 'article', 'media_file', 'user', 'settings'
    resource_id: Mapped[UUID | None]
    changes: Mapped[dict | None]     # JSONB — {"title": {"old": "...", "new": "..."}}
    request_id: Mapped[str | None]   # Correlation с request logs
    ip_address: Mapped[str | None]
```

### Что аудировать

**Обязательно:**

- CRUD статей (создание, публикация, удаление)
- Загрузка/удаление медиафайлов
- Вход/выход (особенно неудачные попытки)
- Изменение ролей пользователей
- Изменение настроек (лимиты загрузки, и т.д.)

**Не аудировать:**

- Чтение/просмотр (это access log)
- Автосохранение черновиков (слишком шумно)

### Имплементация

Явный вызов из сервисного слоя, **не** ORM events:

```python
audit_service.log(db, user_id, action="article.published",
                  resource_type="article", resource_id=article.id,
                  changes={"status": {"old": "draft", "new": "published"}})
```

### Retention

- **Audit log**: 1 год
- **Error log**: 90 дней
- Периодическая очистка — cron или management command

## PW-042-E: Admin UI — мониторинг и аудит

### Где в админке

Новый подраздел в Настройках (по аналогии с «Хранилище»):

```
Настройки → Мониторинг
├── Таб «Система»  — здоровье сервера, диск, память, uptime
├── Таб «Ошибки»   — таблица ошибок API с traceback и resolved
└── Таб «Аудит»    — таблица действий: кто, когда, что
```

### Таб «Система»

- Карточки: uptime, CPU, память, диск (с Progress bar и цветовой индикацией)
- Статус сервисов: API, DB, Storage — online/offline badge
- Версии: Python, бэкенд, БД
- Счётчик ошибок за 24ч (красный badge если > 0)
- Кнопка «Обновить»

### Таб «Ошибки»

- Таблица с пагинацией: дата, уровень, event, path, status_code
- Фильтры: по уровню, resolved/unresolved, по дате
- Раскрытие строки → полный traceback + context JSON
- Кнопка «Resolved» для пометки обработанных
- Счётчик: X ошибок за 24ч / 7д / 30д

### Таб «Аудит»

- Таблица с пагинацией: дата, пользователь, действие, ресурс, IP
- Фильтры: по действию, по пользователю, по дате
- Раскрытие строки → показ `changes` JSONB (diff: old → new)

## AI-Agent Friendly Design

### Принципы логирования для эпохи ИИ

1. **Event names, не сообщения**: `event="article.published"` вместо `msg="Article was published successfully"` — Events
   enumerable, greppable, machine-actionable

2. **Dimensional data как отдельные поля**: `user_id`, `article_id`, `duration_ms` — top-level JSON keys, не вложенные в
   строку

3. **Единые имена полей** across all sources:
   - `timestamp` (ISO 8601, UTC)
   - `level` (debug/info/warning/error/critical)
   - `event` (snake_case)
   - `request_id` (correlation)
   - `service` (api/web/admin)

4. **Контекст ошибок для автоматической отладки**:

   ```json
   {
     "event": "s3_upload_failed",
     "level": "error",
     "bucket": "pw-media",
     "key": "images/2026/03/photo.jpg",
     "error_type": "ClientError",
     "error_code": "AccessDenied"
   }
   ```

   ИИ-агент, видя такой лог, может сразу предложить: «Проверь IAM policy на PutObject permission»

5. **Audit log как data source для ИИ-ассистента**: — «Что изменилось вчера?» → SELECT из audit_log — «Кто удалил статью
   X?» → audit_log WHERE action='article.delete' AND resource_id=X

6. **Error log как замена Sentry для ИИ-агента**: — «Какие ошибки были за последний час?» → SELECT из error_log —
   «Покажи traceback последней 500-ки» → error_log ORDER BY timestamp DESC LIMIT 1 — Агент может сам анализировать
   traceback и предлагать fix

## Что НЕ делаем (осознанно)

| Что                        | Почему нет                                                    |
| -------------------------- | ------------------------------------------------------------- |
| Sentry / внешний SaaS      | Своя таблица ошибок достаточна, данные остаются у нас         |
| ELK Stack / Grafana + Loki | 2-4 GB RAM только на мониторинг — больше, чем само приложение |
| OpenTelemetry + Jaeger     | Distributed tracing для distributed systems, у нас одна VM    |
| Kafka / Celery для логов   | Один сервер, синхронно достаточно                             |
| Per-endpoint метрики       | p50/p95/p99 бессмысленны при десятках RPS                     |
| Полный before/after diff   | Для CMS достаточно логировать action + changed fields         |
| Prometheus counters        | Overkill для блога, `/health` endpoint достаточно             |
