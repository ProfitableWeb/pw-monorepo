# PW-045: Список пользователей — UI/UX + API-интеграция

**Статус**: TODO **Создано**: 2026-03-15 **Приоритет**: Normal **Компонент**: ⚙️ Admin + 🐍 Backend

## Проблема

Раздел «Список пользователей» (`sections/users/assets/UsersList.tsx`) работает на моковых данных (`users.constants.ts`).
Верстка неэффективно расходует вертикальное пространство:

- Статистика занимает 4 отдельных `Card` с `CardHeader` + `CardContent` — много паддингов, вертикали
- Список пользователей уходит за экран, скроллится вся страница целиком
- Нет закреплённого блока фильтров — при скролле кнопки «Пригласить» и «Фильтры» уезжают
- На бэкенде нет admin-эндпоинтов для управления пользователями (только `/users/me` для профиля)

## Цель

Полностью рабочий раздел «Список пользователей»: данные из API, компактная шапка со статистикой, одноэкранный layout с
фиксированной панелью фильтров и скроллом только внутри таблицы/списка.

### PW-045-A: Backend — admin-эндпоинты управления пользователями

**Файлы**:

- `apps/api/src/api/admin/users.py` (новый) — роутер
- `apps/api/src/schemas/users/` (новый пакет) — admin-схемы

**Эндпоинты** (prefix `/admin/users`, зависимость `get_current_admin`):

| Метод  | Путь                | Описание                                        |
| ------ | ------------------- | ----------------------------------------------- |
| GET    | `/admin/users`      | Список с пагинацией, фильтрами, поиском         |
| GET    | `/admin/users/{id}` | Детали пользователя                             |
| PATCH  | `/admin/users/{id}` | Обновить профиль/роль/статус                    |
| DELETE | `/admin/users/{id}` | Деактивировать (soft delete: `is_active=False`) |

#### GET /admin/users — query-параметры

```
?page=1&limit=20                      # Пагинация
&search=иванов                        # Поиск по name/email
&role=admin,editor                    # Фильтр по ролям (через запятую)
&status=active|inactive|invited       # Фильтр по статусу
&sort=created_at&order=desc           # Сортировка
```

#### Схемы ответов

```python
# schemas/users/admin.py

class UserAdminBriefResponse(BaseModel):
    """Элемент списка."""

    name: str
    email: str

    role: str                    # admin/editor/author/viewer
    is_active: bool
    created_at: datetime
    last_login_at: datetime | None
    articles_count: int          # COUNT(articles)

class UserAdminDetailResponse(UserAdminBriefResponse):
    """Детальная карточка."""
    oauth_providers: list[str]
    has_password: bool
    updated_at: datetime

class UserAdminUpdateRequest(BaseModel):
    """Обновление пользователя администратором."""
    name: str | None = None
    email: str | None = None
    role: str | None = None      # валидация: admin/editor/author/viewer
    is_active: bool | None = None

class UserListMeta(BaseModel):
    """Метаданные списка (статистика)."""
    total: int
    active: int
    inactive: int
    by_role: dict[str, int]      # {"admin": 1, "editor": 2, ...}
```

#### Безопасность

- Только `admin` может менять роли и деактивировать (не `editor`)
- Нельзя деактивировать самого себя
- Нельзя понизить роль последнего admin
- `articles_count` — через `func.count()` в JOIN, не через N+1

## Критерии приемки

- [ ] `GET /admin/users` возвращает список с пагинацией, поиском, фильтрами по роли/статусу
- [ ] `GET /admin/users/{id}` возвращает детали пользователя с OAuth-провайдерами
- [ ] `PATCH /admin/users/{id}` позволяет менять роль и деактивировать (только admin)
- [ ] `DELETE /admin/users/{id}` — soft delete (is_active=False), нельзя себя
- [ ] Статистика (total, active, invited, articles) приходит из API, не считается на клиенте
- [ ] UI: статистика в одну строку, компактно
- [ ] UI: одноэкранный layout, скролл только внутри списка пользователей
- [ ] UI: панель фильтров закреплена, не уезжает при скролле
- [ ] Поиск по имени/email работает через API с debounce
- [ ] `last_login_at` отображается вместо мокового `lastActive`
- [ ] Моковые данные удалены
- [ ] Alembic-миграция для `last_login_at`

## Что НЕ делаем

| Что                                          | Почему                                              |
| -------------------------------------------- | --------------------------------------------------- |
| Приглашения (invite flow)                    | Отдельная задача — нужна email-рассылка, токены     |
| Модал создания пользователя                  | Пользователи регистрируются через OAuth, не вручную |
| Детальная страница профиля пользователя      | Отдельная задача, если понадобится                  |
| Bulk-операции (массовое изменение ролей)     | Оптимизация на будущее                              |
| URL для подразделов (users/list, users/team) | Явно отложено в PW-044                              |

## Риски

| Риск                                     | Митигация                                             |
| ---------------------------------------- | ----------------------------------------------------- |
| N+1 при подсчёте articles_count          | JOIN с func.count() + group_by в SQL-запросе          |
| Последний admin понижает себя            | Проверка в сервисе: count(admin) > 1 перед изменением |
| Большой список при тысячах пользователей | Пагинация на бэкенде, limit по умолчанию 20           |
