# PW-038: Article CRUD API — Бэкенд для редактора статей

**Статус**: Планирование **ID задачи**: PW-038 **Контекст**: ProfitableWeb API (`apps/api`) **Зависит от**: PW-037
(фронтенд Article Workbench)

## 1. Цель

Реализовать полный CRUD API для управления статьями из админ-панели. Расширить модель `Article` SEO-полями и
артефактами, добавить систему ревизий, создать защищённые admin-эндпоинты для создания, редактирования, публикации и
удаления статей.

## 2. Scope

### Включено

- Расширение модели `Article` (SEO-поля, артефакты, формат контента)
- Новая модель `ArticleRevision` (история изменений)
- Admin CRUD эндпоинты (`/api/admin/articles`)
- Управление статусом (publish/unpublish)
- Ревизии (список, просмотр, восстановление)
- Автогенерация slug (транслитерация кириллицы)
- Расчёт reading_time на сервере
- Admin-эндпоинты для тегов и категорий (список, создание)
- Alembic-миграция

### Исключено

- Загрузка изображений / S3 (image_url остаётся строковым полем для ручного ввода URL)
- Медиатека / галерея (отдельная задача)
- AI-генерация (артефакты, конвертация форматов) — будущая задача
- Связь с Research Workspace (PW-039) — только поле `provenance` в артефактах

## 3. Изменения модели данных

### 3.1. Расширение `Article` (models/article.py)

Новые поля (все nullable, обратно совместимы):

| Поле               | Тип                    | Описание                                                                     |
| ------------------ | ---------------------- | ---------------------------------------------------------------------------- |

| `focus_keyword`    | `String(200)`          | Основное ключевое слово                                                      |
| `seo_keywords`     | `JSON`                 | Массив дополнительных ключевых слов `["kw1", "kw2"]`                         |

| `robots_no_index`  | `Boolean`              | `default=False`                                                              |
| `robots_no_follow` | `Boolean`              | `default=False`                                                              |

**Маппинг фронтенд → бэкенд:**

- `ArticleFormData.h1` → `Article.title` (уже существует)

### 3.2. Структура `artifacts` (JSON)

```json
{
  "self_check": {
    "enabled": true,
    "items": [{ "id": "sc-1", "question": "...", "answer": "..." }]
  },
  "sources": {
    "enabled": true,
```

JSONB вместо отдельных таблиц — артефакты всегда загружаются/сохраняются вместе со статьёй, независимое обращение к ним
не нужно.

### 3.3. Новая модель `ArticleRevision`

```
models/article_revision.py

class ArticleRevision(UUIDMixin, Base):
    __tablename__ = "article_revisions"

    article_id: UUID (FK articles.id, CASCADE)
    content: Text           # Снимок HTML/Markdown
    content_format: Enum    # Формат на момент ревизии
    summary: String(500)    # Описание изменений
    author_id: UUID (FK users.id, SET NULL)
    created_at: datetime    # Не нужен updated_at — ревизии иммутабельны
```

Ревизия создаётся автоматически при обновлении поля `content` через PATCH.

## 4. API-эндпоинты

### 4.1. Admin Articles — `/api/admin/articles`

Все эндпоинты защищены `get_current_admin` (роли: `admin`, `editor`).

| Метод    | Путь                                                  | Описание                         |
| -------- | ----------------------------------------------------- | -------------------------------- |
| `POST`   | `/api/admin/articles`                                 | Создать статью (draft)           |
| `GET`    | `/api/admin/articles`                                 | Список всех статей (все статусы) |
| `GET`    | `/api/admin/articles/{id}`                            | Получить по UUID (полные данные) |
| `PATCH`  | `/api/admin/articles/{id}`                            | Частичное обновление             |
| `DELETE` | `/api/admin/articles/{id}`                            | Удаление (мягкое → archived)     |
| `POST`   | `/api/admin/articles/{id}/publish`                    | Опубликовать (немедленно)        |
| `POST`   | `/api/admin/articles/{id}/schedule`                   | Запланировать на дату            |
| `POST`   | `/api/admin/articles/{id}/unpublish`                  | Снять с публикации → draft       |
| `GET`    | `/api/admin/articles/{id}/revisions`                  | Список ревизий                   |
| `GET`    | `/api/admin/articles/{id}/revisions/{rev_id}`         | Содержимое ревизии               |
| `POST`   | `/api/admin/articles/{id}/revisions/{rev_id}/restore` | Восстановить                     |

#### POST /api/admin/articles

```
Body: ArticleCreateRequest
  title: str                    # H1 (обязательное)
  subtitle: str | None
  slug: str | None              # Если пусто — автогенерация из title
  status: "draft" | "scheduled" = "draft"
  published_at: datetime | None       # Дата публикации (для scheduled — обязательно)
  content: str = ""
  content_format: "html" | "markdown" = "html"
  excerpt: str = ""
  category_id: str              # UUID категории (обязательное)
  tags: list[str] = []          # Имена тегов — создаются если не существуют
  image_url: str | None
  image_alt: str | None
  layout: str = "three-column"
  # SEO
  meta_title: str | None
  meta_description: str | None
  focus_keyword: str | None
  seo_keywords: list[str] = []
  schema_type: str = "BlogPosting"
  canonical_url: str | None
  og_title: str | None
  og_description: str | None
  og_image: str | None
  robots_no_index: bool = False
  robots_no_follow: bool = False
  # Артефакты
  artifacts: dict | None

Response: ApiResponse[ArticleAdminResponse]   # 201
```

Логика:

- Статус = DRAFT
- `author_id` = текущий пользователь
- Автогенерация slug если не передан (транслитерация title → slugify)
- Проверка уникальности slug (409 Conflict при коллизии)
- Расчёт `reading_time` из content
- Создание начальной ревизии «Создание статьи»

#### GET /api/admin/articles

```
Query params:
  page: int = 1
  limit: int = 20
  status: "draft" | "published" | "archived" | "scheduled" | None  # фильтр по статусу
  category: str | None          # slug категории
  search: str | None            # поиск по title/excerpt
  sort_by: str = "updated_at"   # сортировка (updated_at, created_at, title, published_at, views)
  order: str = "desc"
  author_id: str | None         # UUID автора

Response: ApiResponse[list[ArticleAdminListItem]] + meta
```

Отличия от публичного `GET /api/articles`:

- Видны ВСЕ статусы (не только PUBLISHED)
- Сортировка по `updated_at` по умолчанию (а не published_at)
- Фильтр по `status` и `author_id`
- Расширенные данные (статус, даты create/update, автор)

#### PATCH /api/admin/articles/{id}

```
Body: ArticleUpdateRequest (все поля опциональные)
  title, subtitle, slug, status, published_at,
  content, content_format, excerpt,
  category_id, tags, image_url, image_alt, layout,
  meta_title, meta_description, focus_keyword, seo_keywords,
  schema_type, canonical_url, og_title, og_description, og_image,
  robots_no_index, robots_no_follow, artifacts

Response: ApiResponse[ArticleAdminResponse]
```

Логика:

- Обновляет только переданные поля (partial update)
- Если изменился `content` — создать новую ревизию
- Если изменился `content` — пересчитать `reading_time`
- Если изменился `slug` — проверить уникальность
- Теги: заменяет полный список (переданные имена → find-or-create)

#### POST /api/admin/articles/{id}/publish

```
Response: ApiResponse[ArticleAdminResponse]
```

Логика:

- Валидация: title, content, excerpt, category_id обязательны
- `status` = PUBLISHED
- `published_at` = now() (если первая публикация; при повторной — не менять)

#### POST /api/admin/articles/{id}/schedule

```
Body: { published_at: datetime }    # Обязательное — дата в будущем

Response: ApiResponse[ArticleAdminResponse]
```

Логика:

- Валидация: title, content, excerpt, category_id обязательны (как при publish)
- Валидация: `published_at` > now()
- `status` = SCHEDULED
- `published_at` = переданная дата
- Фоновая задача (или cron) переведёт статью в PUBLISHED по наступлении даты

#### DELETE /api/admin/articles/{id}

```
Query: permanent: bool = false

permanent=false → status = ARCHIVED (мягкое удаление)
permanent=true  → физическое удаление (CASCADE: ревизии, комментарии)

Response: ApiResponse[None]   # 204
```

## 11. Технические решения

### Почему отдельные admin-файлы, а не расширение существующих

- Публичный API (`api/articles.py`, `services/article.py`) фильтрует по PUBLISHED — отдельная логика
- Admin-схемы возвращают расширенные данные (SEO, артефакты, category как объект)
- Разделение ответственности: публичный API не зависит от admin-логики

### Почему JSONB для артефактов

- Загружаются/сохраняются всегда вместе со статьёй
- Не нужны отдельные SQL-запросы к артефактам
- Гибкая структура (новые типы артефактов без миграций)
- Один JSON-столбец vs 4 таблицы + JOIN'ы

### Почему `meta_title` вместо переименования `title`

- Обратная совместимость: публичный API продолжает возвращать `title` (= H1)
- Нет необходимости в миграции данных
- Чистое семантическое разделение: `title` = заголовок для читателей, `meta_title` = SEO

### Ревизии — только content

- Сохраняется снимок `content` + `content_format`, а не всех полей
- Пользователя интересует именно история контента (как в WordPress)
- SEO-поля и метаданные не версионируются — это снижает объём данных
