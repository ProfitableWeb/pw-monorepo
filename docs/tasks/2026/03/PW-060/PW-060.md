## Постановка задачи

### Описание

Добавить сущность «Страница» (Page) по аналогии с WordPress `post_type`. Страницы и статьи используют одну таблицу
`articles`, но различаются полем `type`. Страницы — статический контент (О сайте, Контакты, Политика конфиденциальности
и т.д.), не участвующий в лентах, категориях и RSS.

### Мотивация

Сейчас статические страницы захардкожены в массиве `STATIC_PAGES` на фронтенде (`apps/web/src/app/[slug]/page.tsx`) и
рендерятся как 404. Нужно управление ими через админку: создание, редактирование контента, SEO-настройки, публикация —
весь тот же CRUD, что и для статей, но с другим типом контента.

### Критерии приемки

- [ ] Поле `type` (article | page) добавлено в модель `Article` + миграция
- [ ] Публичный API статей (`GET /api/articles`) возвращает только `type=article`
- [ ] Админ API поддерживает `type` в create/update/response/фильтрации
- [ ] В админке есть раздел «Страницы» (отдельный пункт меню или фильтр)
- [ ] Редактор статей поддерживает создание/редактирование страниц
- [ ] Публичный сайт рендерит страницы по slug (вместо 404 для about, contact и т.д.)
- [ ] `schema_type` для страниц по умолчанию `WebPage`, для статей — `BlogPosting`
- [ ] Страницы не отображаются в RSS, sitemap/articles, лентах категорий

## Архитектурное решение

**Подход**: WordPress-style `post_type` — одна таблица, поле-дискриминатор.

**Почему не отдельная таблица**: Статьи и страницы идентичны по структуре данных (заголовок, контент, SEO, изображения,
layout). Отдельная таблица = дублирование модели, схем, сервисов, CRUD-эндпоинтов. Одна таблица с `type` — минимальные
изменения.

## План реализации

### Фаза 1: Backend — модель и миграция

1. **Модель** (`apps/api/src/models/article.py`):
   - Добавить enum `ArticleType(str, Enum): ARTICLE = "article", PAGE = "page"`
   - Добавить поле `type: Mapped[ArticleType]` с default `ARTICLE`

2. **Миграция** (Alembic):
   - `ALTER TABLE articles ADD COLUMN type VARCHAR(20) NOT NULL DEFAULT 'article'`
   - Индекс на `type` для быстрой фильтрации

### Фаза 2: Backend — сервисы и API

3. **Публичные запросы** (`apps/api/src/services/articles/queries.py`):
   - В `_base_published_query()` добавить `.where(Article.type == ArticleType.ARTICLE)`
   - Новая функция `get_page_by_slug()` — только `type=page`

4. **Админ-сервис** (`apps/api/src/services/articles/admin.py`):
   - Добавить `"type"` в `UPDATABLE_FIELDS`
   - Параметр `type` в `create_article()`
   - Опциональный фильтр `type` в `get_all_articles()`

5. **Схемы** (`apps/api/src/schemas/articles/admin.py`):
   - `type` в `ArticleCreateRequest` (default: `"article"`)
   - `type` в `ArticleUpdateRequest` (optional)
   - `type` в `ArticleAdminResponse`

6. **Эндпоинты**:
   - Админ: передавать `type` в create/update, фильтр в list
   - Публичный: новый `GET /api/pages/{slug}` или использовать общий `GET /api/articles/{slug}` без фильтра по type

### Фаза 3: Admin Frontend

7. **Навигация** (`navigation.constants.ts`):
   - Новый пункт меню «Страницы» в секции «Контент» (или фильтр-переключатель в списке статей)

8. **Список страниц**:
   - Переиспользовать `ArticlesTable` с фильтром `type=page`
   - Упрощённый набор колонок (без категории, просмотров, автора)

9. **Редактор** (Article Workbench):
   - Селектор типа при создании (Статья / Страница)
   - Для страниц: скрыть категории, теги, reading_time
   - Layout по умолчанию: `one-column` для страниц

### Фаза 4: Web Frontend

10. **Роутинг** (`apps/web/src/app/[slug]/page.tsx`):
    - Убрать хардкод `STATIC_PAGES`
    - Добавить проверку `getPageBySlug(slug)` перед категориями
    - Рендерить страницы тем же `ArticlePage` компонентом (или упрощённым `PageComponent`)

11. **SEO**:
    - Страницы: `schema_type = "WebPage"`, без `datePublished`/`author` в JSON-LD
    - Исключить страницы из RSS-фида и sitemap секции articles

## Риски и заметки

- **Slug uniqueness**: Сохранить глобальную уникальность slug (без разделения по type)
- **Обратная совместимость**: default `type=article` — все существующие записи автоматически статьи
- **Категории/теги**: Для страниц — опциональны (можно скрыть в UI, не запрещать в API)
- **Ревизии**: Работают одинаково для обоих типов
