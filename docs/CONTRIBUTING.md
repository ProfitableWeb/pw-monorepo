# Правила участия

Спасибо за интерес к проекту ProfitableWeb! Мы рады любому вкладу.

## Как внести вклад

### 1. Создайте Issue

Перед началом работы создайте issue в репозитории:

- **Баг** — опишите шаги воспроизведения, ожидаемое и фактическое поведение
- **Фича** — опишите предлагаемую функциональность и зачем она нужна
- **Улучшение** — опишите, что хотите изменить и почему

### 2. Форк и ветка

```bash
# Форк репозитория через интерфейс GitVerse
git clone https://gitverse.ru/<ваш-username>/pw-monorepo.git
cd pw-monorepo
git checkout -b feat/PW-XXX-краткое-описание
```

### 3. Настройка окружения

```bash
# JavaScript-зависимости
bun install

# Python-зависимости (для apps/api)
cd apps/api
uv sync
cd ../..
```

### 4. Внесите изменения

Следуйте стандартам кода проекта:

- **Frontend (apps/web)**: SCSS-модули, без Tailwind, без CSS-in-JS
- **Admin (apps/admin)**: Tailwind CSS, Radix UI, Zustand
- **Backend (apps/api)**: сервисный слой, Pydantic-схемы, синхронный SQLAlchemy

### 5. Проверьте код

```bash
# Линтинг
bun turbo lint

# Проверка типов
bun turbo type-check

# Тесты
bun turbo test

# Для backend отдельно
cd apps/api
uv run ruff check
uv run mypy .
uv run pytest
```

### 6. Оформите коммит

Формат коммитов: `type(PW-XXXX): описание на русском`

Типы: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `chore`, `ci`

Примеры:

- `feat(PW-042): добавить фильтрацию статей по тегам`
- `fix(PW-042): исправить пагинацию на мобильных`

### 7. Создайте запрос на слияние

- Заголовок должен кратко отражать суть изменений
- В описании укажите: что сделано, зачем, как протестировано
- Привяжите связанный issue
- Убедитесь, что все проверки проходят (lint, type-check, тесты)

## Стиль кода

- **Язык**: русский для пользовательского контента, комментариев и коммитов
- **Форматирование**: Prettier (конфиг в `.prettierrc.js`)
- **Линтинг**: ESLint для JS/TS, Ruff для Python
- **Типизация**: TypeScript (strict) для фронтенда, mypy для бэкенда

## Процесс ревью

1. Автоматические проверки (lint-staged через Husky)
2. Ревью сопровождающими проекта
3. После одобрения — слияние в `master`

## Сообщение об уязвимостях

Не создавайте публичные issues для уязвимостей безопасности. См. [SECURITY.md](./SECURITY.md) для инструкций по
ответственному раскрытию.

## Кодекс поведения

Участвуя в проекте, вы соглашаетесь соблюдать наш [Кодекс поведения](./CODE_OF_CONDUCT.md).
